# 自动导演统一运行时重构方案

更新日期：2026-04-28

落地说明：本文是自动导演重构总纲，负责确定统一方向和长期边界。第一阶段工程实施应参考 `docs/plans/auto-director-mvp-migration-plan.md`，先完成最小可迁移运行时、最小产物账本、最小控制策略和最小节点契约，再逐步引入更完整的创作质量模块与 LangGraph 编排。

## 1. 背景

当前自动导演相关能力已经覆盖了新建小说、候选方向、宏观规划、角色准备、卷规划、结构化拆章、章节执行、质量修复、任务检查点、失败恢复和已有项目接管等环节。但这些能力在产品体验和后端架构上仍然呈现出明显的割裂感：

- “自动导演创建”更像一条从空白项目开始的顺序生成链路。
- “AI 自动导演接管”更像一条面向已有资产的恢复链路。
- 用户手动修改后，系统很难自然判断应该继续、局部修复、重跑下游，还是暂停等待确认。
- 自动、半自动、手动审核之间的切换主要依赖流程分支和检查点拼接，而不是一个统一的控制权模型。

本方案的目标不是简单整理文件，也不是只把现有流程搬进 LangGraph，而是重建自动导演的核心抽象：让新建、接管、暂停、手动编辑、自动继续、失败恢复都成为同一个导演运行时上的不同入口和控制策略。

## 2. 核心判断

### 2.1 自动导演和接管不应是两条链

自动导演创建和 AI 自动导演接管的本质相同：系统读取当前小说工作区，判断已有资产与缺失资产，选择下一步最合理的生产动作，并按用户授权程度继续推进。

区别只在入口：

- 新建小说：从用户灵感和基础设定开始。
- 接管小说：从已有小说资产开始。
- 手动后继续：从用户刚修改过的资产版本开始。
- 失败后恢复：从最后可信检查点和产物版本开始。

因此，接管不应该维护独立主链。它应该是统一导演运行时中的“工作区分析 + 恢复/继续决策”。

### 2.2 自动和手动不应是流程分叉

自动、半自动、手动审核、只建议不执行，不应该对应多套业务流程。它们应该是同一条导演链上的控制策略。

用户需要的是随时切换控制权：

- AI 可以自动跑到下一个关键检查点。
- 用户可以随时暂停并手动修改。
- 用户修改后，AI 能理解改动影响并继续。
- 系统可以建议下一步，但不强迫用户交出控制权。
- 高风险覆盖、重写、下游重算必须清楚说明影响范围并等待确认。

这要求系统从“流程分支”转向“统一运行时 + 控制策略”。

### 2.3 产物比步骤更重要

当前系统的复杂度很大一部分来自“步骤状态”承担了太多职责。理想情况下，导演运行时应该围绕可复用、可版本化、可审核的小说生产产物推进。

关键产物包括但不限于：

- 书级方向与读者承诺。
- Book Contract。
- 故事宏观规划。
- 角色阵容与关系动力。
- 卷级战略与卷骨架。
- 章节任务单、冲突、揭示、情绪目标和结尾钩子。
- 正文草稿。
- 审核报告与修复计划。
- 连续性记忆与后续生产上下文。

步骤只是生成、验证、更新这些产物的方式。用户手动编辑和 AI 自动生成都应该写入同一套产物体系。

### 2.4 拆书、知识库、世界观和角色控制应成为核心导演资产

当前自动导演更偏向“从用户输入生成小说计划”，但对已有内容资产的吸收还不够深：

- 拆书结果和知识库更多像可选参考，没有充分进入书级方向、卷规划、章节任务单和正文执行的核心决策。
- 世界观在自动导演模式中的参与度偏低，容易导致设定只在背景层存在，没有真正约束冲突、行动、资源、规则和代价。
- 角色控制偏少，角色更像一次性生成的设定清单，而不是贯穿长篇生产的欲望、关系、弧光、当前状态和出场责任。

这三类能力不应该作为后续附加功能处理，而应该在本次统一运行时重构中成为导演可读取、可更新、可校验的核心资产。

特别是当用户没有绑定已有世界观时，自动导演需要考虑是否生成“项目级世界观骨架”。这个步骤不应要求新手先手动设计复杂世界，而应由 AI 根据题材、读者承诺、故事模式和主线冲突自动生成可执行的世界规则，并在后续卷规划和章节执行中持续使用。

### 2.5 质量审核和承诺校验不应阻断整条链

质量审核、承诺兑现检查、伏笔账本、世界观一致性和角色状态检查都很重要，但它们不应被设计成“一处失败，全链停摆”的硬阻断。

更合理的原则是：

- 质量检查优先产出可执行修复建议，而不是只给失败结论。
- 自动修复可以尝试，但默认最多自动重试一次，避免无限循环和成本失控。
- 自动重试仍未通过时，应进入“可解释的人工修复/确认继续”状态。
- 非破坏性问题不应阻止后续链路继续推进，可以标记为待修复风险。
- 高风险问题应暂停对应影响范围，而不是冻结整本书所有生产动作。
- 用户始终可以手动修复、接受风险继续、或选择后续局部重算。

这意味着质量系统应是“弹性护栏”，不是“单点熔断器”。自动导演需要记录问题、解释影响、推荐修复路径，但不能因为一个章节、一个承诺项或一次审核失败，就让用户无法继续后续创作。

### 2.6 新增处理模块必须低成本接入

本次重构还应解决后续扩展问题。自动导演未来一定会继续增加能力，例如世界观治理、读者画像校准、开篇留存强化、中段疲劳检测、商业化卖点检查、角色关系推进、题材规则校验、付费转化节奏评估等。

如果每新增一个能力都要改主 service、改阶段分支、改接管逻辑、改创作中枢工具调用，系统会很快再次变乱。因此统一运行时必须提供标准扩展点：

- 新增处理节点：用于生成、分析、修复或同步某类导演产物。
- 新增质量门：用于检查承诺、节奏、角色、世界观、伏笔、留存等问题。
- 新增上下文来源：用于接入拆书、知识库、用户资料、参考作品或外部分析结果。
- 新增产物类型：用于被账本版本化、依赖追踪、过期识别和局部重算。
- 新增创作中枢动作：用于把导演能力作为工具暴露给对话式中枢。
- 新增事件投影：用于让前端和任务中心理解模块进度，而不是直接读内部实现。

新增模块应通过“能力注册 + 节点契约 + 产物账本 + 策略层”接入，而不是通过主流程里的硬编码分支接入。主链越稳定，后续能力扩展越快，也越不容易破坏手动/自动切换和接管恢复。

## 3. 当前模块颗粒度与串联方式

本节记录当前实现的真实颗粒度，便于后续重构时判断哪些模块可以包裹迁移，哪些模块需要拆分或下沉。

### 3.1 当前入口颗粒度

当前自动导演对外入口主要分成三类：

- 候选方向入口：生成候选、重新生成、修补单个候选、重做标题。
- 确认与新建入口：确认候选后创建小说，并启动后台导演主链。
- 继续与接管入口：从已有任务检查点继续，或从已有小说状态发起接管。

入口层的颗粒度相对清晰，但它们进入后端后会迅速汇聚到同一个大型服务，由大型服务继续判断当前应该走候选、主链、接管、恢复、标题修复还是自动执行。

### 3.2 当前主编排颗粒度

当前主编排集中在 `NovelDirectorService`。它承担了过多职责：

- 作为路由入口背后的应用服务。
- 创建和复用 workflow task。
- 构造 seed payload、director session、resume target。
- 调度后台任务。
- 串联 story macro、角色准备、卷规划、结构化拆章、自动执行。
- 处理 continue、takeover、recovery、title repair、auto approval 和高内存保护。

它目前更像“自动导演总控台”，而不是一个薄的 orchestrator。后续重构应把它收敛为兼容旧 API 的门面，核心运行时下沉到独立 Director Runtime。

### 3.3 当前阶段模块颗粒度

当前各阶段模块颗粒度并不一致：

| 当前模块 | 当前职责颗粒度 | 判断 |
| --- | --- | --- |
| Candidate Stage | 候选方向生成、framing 对齐、候选修补、标题组生成、候选选择检查点 | 颗粒度相对合理，但仍直接依赖 workflow task 状态 |
| Story Macro Phase | 故事宏观规划、约束引擎、Book Contract | 阶段清晰，适合包装成未来节点 |
| Pipeline Phases | 角色准备与卷战略放在同一文件中 | 文件职责偏混合，应拆成角色治理与卷规划节点 |
| Structured Outline Phase | beat sheet、chapter list、chapter detail bundle、chapter sync、front10 checkpoint | 颗粒度偏粗，是后续拆节点的重点 |
| Auto Execution Runtime | 执行范围解析、章节 pipeline job 循环、审核修复检查点、完成/暂停状态同步 | 更像一个子运行时，应保留为章节执行节点背后的执行器 |
| Takeover Runtime / Execution / Reset / Continue | 接管分析、重启当前步骤、下游重置、继续已有自动执行 | 当前像第二条链路，应并入 Workspace Analyzer 与 Runtime recovery |
| Workflow Service | 任务状态、检查点、seed payload、恢复标记、任务中心投影 | 是状态承载层，但不应继续承载过多业务真相 |

### 3.4 当前串联方式

当前串联方式大致如下：

```text
前端自动导演入口
  ↓
novelDirector routes / novelWorkflows routes
  ↓
NovelDirectorService
  ↓
候选阶段：CandidateStageService
  ↓
确认候选：createNovel + bootstrapTask + scheduleBackgroundRun
  ↓
runDirectorPipeline
  ↓
story_macro
  ↓
character_setup
  ↓
volume_strategy
  ↓
structured_outline
  ↓
front10_ready checkpoint
  ↓
autoExecutionRuntime.runFromReady（可选）
  ↓
chapter pipeline / quality repair / completed or paused checkpoint
```

阶段之间主要通过以下方式交互：

- 函数调用直接传递 `taskId`、`novelId`、request、workspace。
- callbacks 回写任务进度。
- workflow checkpoint 控制暂停与继续。
- seed payload 保存 director session、resume target、candidate batch、auto execution state 等上下文。
- 接管和恢复模块读取已有任务、已有小说资产、pipeline job、checkpoint，再决定回到主链或自动执行子运行时。

这套方式能跑起来，但缺少统一节点契约。阶段之间不是通过“输入产物、输出产物、下一步决策”串联，而是通过服务调用、任务状态、seed payload、checkpoint 和若干恢复函数共同拼接。

### 3.5 当前颗粒度问题

当前最主要的问题不是“阶段不够多”，而是颗粒度层级混在一起：

- 应用服务、领域阶段、任务状态、恢复策略、后台调度混在 `NovelDirectorService`。
- 结构化拆章阶段过大，内部已经包含多个天然节点。
- 角色只是准备阶段，不是贯穿后续执行的长期治理模块。
- 世界观、拆书、知识库没有被纳入主链节点契约。
- 新增处理能力缺少能力注册和节点契约，后续扩展容易继续堆到主 service 或旧分支里。
- 接管逻辑单独演化，导致它需要猜测主链状态，而不是从统一运行时读取状态。
- 自动执行子运行时有价值，但和导演主链之间缺少清晰边界。

后续重构不建议继续按文件名小修小补，而应先确定节点层级：入口动作、导演运行、阶段节点、子执行器、产物账本、事件投影分别负责什么。

## 4. 当前功能设计的网文创作缺陷与优化点

本节从中文长篇网文创作角度审视当前功能设计。当前系统已经有不少高价值能力：书级候选、主钩子、前 30 章承诺、Book Contract、故事宏观规划、分卷策略、章节任务单、场景卡、章节审校修复、伏笔账本、角色资产、动态角色、角色资源和章节状态同步。问题不在于“完全没有”，而在于这些能力尚未被统一导演链稳定组织成网文生产闭环。

### 4.1 读者承诺没有形成全链路闭环

当前已有目标读者、卖点、前 30 章承诺、主钩子、chapter3 / chapter10 / chapter30 payoff 等概念，但它们更多分散在候选、Book Contract、角色准备和章节上下文里。

缺陷：

- 书级承诺没有被明确拆成“每卷承诺、每个节奏段承诺、每章留存动作”。
- 前 30 章承诺虽然存在，但缺少可视化或结构化追踪，难以判断第 3、10、30 章是否真的兑现。
- 章节执行重视任务单和钩子，但缺少“这一章对读者承诺的贡献”这类硬性检查。

建议：

- 将读者承诺纳入 Artifact Ledger，成为可追踪产物。
- 每个卷、节奏段、章节都声明对哪个承诺负责。
- 审核阶段增加“承诺兑现度”和“追读理由”维度。
- 自动导演下一步推荐应优先修复承诺断裂，而不是只看流程是否完成。

### 4.2 开篇抓力仍缺少专门节点

当前候选阶段会生成 hookStrategy，章节写作也要求开头迅速进入情境、结尾形成钩子。但开书最关键的前 1-3 章仍然缺少专门的“开篇抓力设计”节点。

缺陷：

- 第一章没有单独承担“让读者立刻知道为什么要追”的设计检查。
- 前 3 章缺少明确的异常开局、主角处境、第一压力、第一小兑现。
- 新手用户容易得到完整规划，但开篇仍然可能不够有点击后留存力。

建议：

- 在 Book Contract 后增加 Opening Hook Plan，或将其作为前期章节 blueprint 的特殊节点。
- 明确第 1 章必须交付：异常情境、主角被困、即时压力、可感知卖点、章末追读钩子。
- 第 2-3 章必须完成第一轮压力升级或小兑现，而不是继续铺设。
- 自动审核增加“前 3 章留存风险”专项。

### 4.3 章节任务单已经存在，但还可以更像“留存合同”

当前章节任务单、boundary、exclusiveEvent、endingState、nextChapterEntryState、sceneCards 已经很强，是一个好基础。

缺陷：

- 任务单偏向“本章要发生什么”，还不够强调“读者为什么看完还想下一章”。
- 结尾钩子存在要求，但没有被细分为威胁、揭示、选择、误会、奖励未完全兑现、关系变化等类型。
- 场景卡关注职责和边界，但对情绪曲线、爽点位置、悬念释放量的控制还不够显式。

建议：

- 将章节任务单升级为 Chapter Retention Contract。
- 每章明确：本章新信息、本章变化、本章小回报、本章未解压力、章末钩子类型。
- 场景卡增加“情绪/爽点位置”与“读者获得感”检查。
- 审核阶段检测章节是否只完成事件，却没有形成追读价值。

### 4.4 角色功能强，但“角色驱动剧情”还不够硬

当前系统已有角色阵容、角色资产、角色动态、关系阶段、角色资源、事件流等能力，这些是很好的底座。

缺陷：

- 自动导演主链中，角色准备之后的后续阶段没有强制让角色欲望、选择和关系张力持续驱动剧情。
- 角色容易被当作章节参与者，而不是每章冲突的发动机。
- 主角阶段性欲望、误区、代价和变化路径没有成为每章必查项。
- 反派或阻力系统的主动推进不足时，章节可能变成主角一路处理任务，而非被持续压迫。

建议：

- Character Governance 不只生成角色，还要维护每个阶段的角色责任。
- 每个章节任务单必须说明关键角色在本章带来的冲突、信息、情绪或选择。
- 每卷检查主角状态变化、关系变化、对手压力是否足够。
- 审核阶段加入“角色是否主动推动局面，而不是被剧情摆放”。

### 4.5 世界观参与不足会削弱冲突成本

当前前端允许绑定世界观，章节上下文也有 world rules，但自动导演主链里世界观更像边界资料，而不是冲突机制。

缺陷：

- 没有绑定世界观时，系统没有明确生成项目级世界规则。
- 绑定世界观后，系统未必会把世界规则转化为行动代价、资源限制、制度压力、空间障碍或信息禁区。
- 世界观不参与冲突设计时，章节事件容易变成通用剧情，缺少题材特异性。

建议：

- 增加 World Architect 节点，按题材决定轻量或重型世界骨架。
- 世界观输出必须能转化为：规则、代价、资源、禁区、组织、地点、阶段性揭示。
- 章节任务单必须说明本章使用了哪个世界规则制造冲突或限制。
- 审核阶段检测“设定是否只被提到，没有参与戏剧行动”。

### 4.6 拆书和知识库没有深度参与主决策

当前续写模式可以选择知识库文档和拆书结果，文案也说明会注入后续规划和章节生成。但从导演重构角度看，它们仍需要升级为主链资产。

缺陷：

- 拆书结果更像上下文增强，不像会改变候选方向、卷节奏、章节任务的决策输入。
- 知识库资料缺少“硬约束 / 软参考 / 风格参考 / 禁区”的分层。
- 参考作品的优势没有被转译成当前项目可执行的结构策略。

建议：

- Source and Knowledge Orchestrator 负责把拆书与知识库转成导演资料包。
- 候选、Book Contract、分卷、章纲、正文阶段读取不同粒度的资料。
- 拆书输出应转化为：结构节奏、爆点分布、关系推进、悬念释放、禁用套路。
- 续写和原创都可以使用资料包，但原创应避免照搬设定和剧情。

### 4.7 中段续航与反复升级需要更强的控制

当前分卷策略已经关注前期抓力、中段续航、后段升级空间，但后续章节生成链路仍需更显式地防止中段变成重复任务。

缺陷：

- 分卷策略到章节任务之间缺少“升级梯度”硬约束。
- 章节列表虽然要求新增推进，但没有形成跨 10-30 章的压力曲线检查。
- 中段最容易出现的同质冲突、重复对话、反复铺垫，需要在导演层预防。

建议：

- 增加 Escalation Ladder 产物，记录压力、对手、风险、奖励和秘密揭示如何升级。
- 每个节奏段必须说明比上一个节奏段升级在哪里。
- 审核不只看单章，还要做章节窗口审查，例如最近 5 章是否同质。

### 4.8 审核修复偏单章，缺少跨章诊断

当前章节审核修复已经有 coherence、repetition、pacing、voice、engagement 等维度，单章质量基础不错。

缺陷：

- 缺少对最近多章的节奏、承诺、角色、世界规则、伏笔兑现进行窗口级审查。
- 自动修复更多是修当前章，较少触发“上游任务单需要调整”或“下游章节需要重算”。
- 如果一个问题来自卷规划或角色动机，单章修文容易治标不治本。

建议：

- 增加 Rolling Window Review：最近 3 / 5 / 10 章窗口审查。
- 审核输出应能指向问题来源：正文问题、任务单问题、卷规划问题、角色治理问题、世界规则问题。
- 修复策略应允许选择：修正文、修任务单、修上游产物、重算下游。
- 质量问题应进入“修复票据 + 影响范围”模型，默认不把整条链标记为失败。
- 自动修复最多尝试一次；仍未通过时转为人工修复、接受风险继续或后续局部重算，而不是无限重试或全局停止。

### 4.9 新手视角仍需要更强的“下一步解释”

当前前端已有不少友好文案，但自动导演和接管仍然容易暴露工程化阶段概念。

缺陷：

- 用户看到“接管”“重跑当前步”“继续已有进度”时，仍需要理解背后的流程风险。
- 自动化执行范围、审批点和下游影响还不够像“写作助理解释”，更像“任务系统解释”。
- 新手最需要的是系统告诉他“现在最该做什么，为什么，做了会影响哪里”。

建议：

- 所有导演动作都配套 `explain_director_next_action`。
- 工作区分析结果默认输出推荐动作，而不是让用户先选 phase 和 strategy。
- 高级参数保留，但默认走 AI 推荐。

### 4.10 值得优先优化的排序

若按“对新手完成整本书”和“网文留存质量”的综合收益排序，建议优先级如下：

1. 统一运行时与工作区分析：先解决多链路割裂。
2. 读者承诺闭环：把书级承诺贯穿卷、章、审核。
3. 世界观与资料包主链化：让拆书、知识库、世界规则真正参与决策。
4. 角色治理：让角色欲望、关系和对手压力持续驱动剧情。
5. 开篇抓力专项：强化前 1-3 章留存。
6. 章节留存合同：升级现有任务单。
7. 跨章窗口审查：防止中段疲软和重复。
8. 创作中枢接入：先解释与控制，再开放高风险自动执行。

## 5. 目标体验

### 5.1 新建小说体验

用户从一个模糊想法开始时，系统应该自动完成低认知负担的引导：

1. 理解用户灵感。
2. 生成可选择的书级方向。
3. 推荐最适合新手完成的方向。
4. 建立书级承诺和长期生产计划。
5. 逐步生成角色、卷规划、章纲和正文。
6. 在关键节点给出清晰确认，而不是要求用户理解专业创作流程。

### 5.2 接管已有小说体验

用户在已有小说中启用 AI 自动导演时，系统应该先分析当前工作区，再给出自然语言建议：

> 已识别到当前小说已有基础设定、主要角色和前 8 章正文。建议先补齐第 1 卷节奏与第 9-20 章任务单，再继续自动生成正文。

用户不应该优先面对 start phase、entry step、strategy 这类工程化选择。系统可以保留高级选项，但默认体验应是“AI 识别现状并推荐下一步”。

### 5.3 手动和自动切换体验

用户应该可以在任何阶段执行以下动作：

- 暂停自动导演。
- 手动修改当前产物。
- 让 AI 评估修改影响。
- 继续下一步。
- 只运行下一步。
- 自动运行到下一个检查点。
- 切换为全自动执行某个范围。
- 回退到上一个可信版本。

切换时系统要解释：

- 当前使用哪些已确认资产。
- 哪些下游产物可能过期。
- 下一步会生成或修改什么。
- 是否会覆盖用户手写内容。

## 6. 目标架构

### 6.1 统一导演运行时

目标是建立一个统一的 Director Runtime，承接所有入口：

- 从灵感创建。
- 从已有小说接管。
- 从暂停点继续。
- 从失败点恢复。
- 从用户手动编辑后继续。
- 从指定范围进入自动执行。

运行时不直接等同于某个具体框架。LangGraph 可以作为阶段图和 interrupt/resume 的执行外壳，但业务核心应由稳定的导演领域模块承载。

建议的架构分层：

```text
入口适配层
新建小说 / 接管已有小说 / 任务继续 / 手动编辑后继续
        ↓
工作区分析层
识别已有资产、缺失资产、过期资产、可继续位置、风险范围
        ↓
导演编排层
根据统一阶段图和当前状态决定下一步
        ↓
策略层
决定自动继续、暂停确认、只建议、自动修复、禁止覆盖
        ↓
产物层
统一保存、读取、版本化、比较和回滚导演产物
        ↓
执行层
调用 prompt、领域服务、章节 pipeline、审核修复服务
```

### 6.2 统一阶段图

阶段图应描述小说从灵感到完整生产的主路径，但每个阶段都允许从已有资产进入。

建议的阶段顺序：

1. 灵感理解与项目定向。
2. 候选书级方向。
3. Book Contract。
4. 故事宏观规划。
5. 角色阵容与关系动力。
6. 卷级战略。
7. 结构化拆章。
8. 章节任务单。
9. 正文生成。
10. 审核与修复。
11. 连续性更新。
12. 下一步推荐或继续执行。

阶段图应支持非线性恢复：

- 已有 Book Contract 时，可以从故事宏观规划继续。
- 已有卷规划但缺少章节任务单时，可以从结构化拆章继续。
- 已有正文但缺少连续性状态时，可以先补齐连续性分析。
- 用户手动改了前文时，可以局部标记下游章纲或正文为需要复核。

### 6.3 产物账本

自动导演应建立统一的产物账本，而不是让产物状态散落在任务 payload、章节记录、卷规划、检查点和接管快照中。

产物账本负责：

- 记录每个导演产物的当前版本。
- 记录产物来自 AI 生成、用户手动编辑、自动修复还是导入。
- 记录产物之间的依赖关系。
- 支持判断下游产物是否可能过期。
- 支持接管时快速识别已有工作区状态。
- 支持失败恢复、版本回退和局部重算。

当前阶段不要求定义具体字段，但后续实现时应优先保证产物账本能回答三个问题：

1. 当前小说已经有哪些可信产物？
2. 下一步最应该补齐或更新哪个产物？
3. 如果继续自动执行，会影响哪些已有产物？

### 6.4 工作区分析器

工作区分析器是统一新建、接管和手动继续的关键模块。

它负责读取当前小说状态，并输出面向导演运行时的判断：

- 当前小说处于哪个生产阶段。
- 哪些关键产物缺失。
- 哪些产物存在但可能过期。
- 哪些用户手写内容需要保护。
- 是否可以直接继续自动生成。
- 是否需要先做补齐、修复、同步或人工确认。
- 推荐的下一步动作是什么。

这里应遵守 AI-first 规则：涉及创作意图、结构判断、质量判断、改动影响的识别，应优先由 AI 产生结构化判断。确定性代码只负责安全校验、状态约束和执行保护。

### 6.5 控制策略层

控制策略层负责把“自动/手动/半自动”从业务流程中抽离出来。

它应该统一处理：

- 是否自动进入下一阶段。
- 是否在当前检查点暂停。
- 是否允许自动应用 AI 产物。
- 是否允许覆盖或重算用户编辑过的内容。
- 是否允许自动审核和自动修复。
- 当前执行范围是下一步、前若干章、某一卷还是整本书。
- 失败后是自动重试、暂停、跳过，还是要求用户确认。

这样，用户切换自动与手动时，不需要换一条链路，只需要改变当前导演运行的控制策略。

质量门也应由控制策略层统一管理，不应散落在各阶段里自行决定是否终止全局流程。建议把审核、承诺校验、伏笔校验、世界观一致性校验和角色状态校验分成以下结果：

- `passed`：检查通过，可以继续。
- `repairable`：发现问题，可以自动修复一次。
- `needs_manual_repair`：自动修复失败或不适合自动改写，需要用户手动处理。
- `continue_with_risk`：问题不影响下一步推进，但需要记录风险和后续修复建议。
- `blocked_scope`：只暂停受影响范围，例如当前章、当前卷或某个下游产物，不冻结整本书运行。

默认策略应是：质量失败先生成修复计划，自动修复最多重试一次；如果仍未通过，进入可解释的人工修复或带风险继续状态。这样可以避免一个审核点、一个承诺项或一个章节问题拖死后续所有创作。

### 6.6 统一事件流

自动导演应通过标准事件向任务中心和前端同步，而不是让各阶段直接拼 UI 文案或自行维护进度解释。

推荐事件类型：

- 导演运行开始。
- 阶段开始。
- 产物生成。
- 产物等待确认。
- 产物被用户修改。
- 自动审批通过。
- 需要人工确认。
- 正在自动执行章节。
- 审核发现问题。
- 修复票据生成。
- 修复计划生成。
- 带风险继续。
- 阶段完成。
- 运行失败。
- 运行恢复。
- 运行完成。

前端和任务中心再根据事件生成用户可读文案。这样可以避免后端流程、任务状态和 UI 解释互相缠绕。

### 6.7 核心创作资产增强方向

统一运行时应把拆书、知识库、世界观和角色控制纳入主链，而不是让它们停留在外围参考层。

#### 6.7.1 拆书与知识库参与

拆书和知识库需要从“可被引用的资料”升级为“导演决策输入”。

在新建、接管和继续执行时，系统应能判断：

- 当前项目是否绑定了拆书结果、知识文档、续写分析或参考作品分析。
- 哪些资料对书级方向、题材承诺、角色关系、世界规则、章节任务最有价值。
- 当前阶段需要读取完整资料、摘要资料、局部片段，还是只读取结构化结论。
- 资料中的哪些内容应成为硬约束，哪些只是风格或灵感参考。

拆书与知识库至少应进入这些阶段：

- 候选书级方向：帮助生成更贴近目标品类和参考作品优点的方向。
- Book Contract：固化读者承诺、市场亮点、禁区和差异化。
- 故事宏观规划：提供结构参考、爽点节奏、反套路空间。
- 卷规划：参考长线升级、关系推进、悬念释放和阶段 payoff。
- 章节任务单：提取可执行的冲突、信息释放、情绪目标和结尾钩子。
- 正文执行：提供风格、设定、专业知识和连续性支持。

#### 6.7.2 世界观参与

世界观不应只作为背景设定。对长篇小说而言，世界观应该回答：

- 这个世界的规则如何制造冲突。
- 主角行动会受到哪些资源、代价、制度、空间、阶层或力量体系限制。
- 哪些设定会在前中后期形成递进式揭示。
- 哪些世界规则会影响角色选择和章节事件。
- 哪些设定不能随意改动，否则会破坏连续性。

如果用户已绑定已有世界观，自动导演应分析并提取可执行规则；如果用户没有绑定世界观，系统应根据项目需要自动生成项目级世界观骨架。

世界观生成不一定对所有题材都是重型步骤。现实题材、都市情感、职场、悬疑等项目也可能只需要轻量世界约束，例如行业规则、生活空间、社会关系、组织结构、案件规则或资源边界。玄幻、科幻、无限流、游戏、架空等题材则需要更完整的规则、力量体系和阶段性揭示计划。

#### 6.7.3 角色控制

角色控制需要从“生成角色卡”升级为“长期角色治理”。

自动导演应持续跟踪：

- 主角当前欲望、压力、恐惧、误区和阶段目标。
- 主要配角的功能、关系张力、变化方向和出场责任。
- 反派或阻力系统的主动推进能力。
- 角色关系在每卷、每若干章中的推进节奏。
- 当前章节中每个关键角色应该带来的冲突、信息、情绪或选择。
- 角色是否开始重复、变工具人、失去动机或违背前文状态。

章节执行不应只知道“有哪些角色”，而应知道“这些角色在当前章节为什么必须出现，以及出现后要改变什么”。

### 6.8 可扩展模块体系

统一导演运行时不应只服务当前这些阶段，还应为未来能力增长预留清晰扩展方式。

建议把可扩展能力分成几类：

- 生产型节点：生成或更新某类产物，例如世界观骨架、角色阶段状态、章节任务单、开篇钩子计划。
- 分析型节点：读取已有内容并输出判断，例如接管分析、手动编辑影响分析、最近 5 章节奏诊断。
- 审核型节点：输出质量报告、风险等级、修复建议和影响范围。
- 修复型节点：基于审核结果执行一次局部修复，并把结果写回产物账本。
- 同步型节点：把正文变化同步到连续性记忆、伏笔账本、角色状态或世界观状态。
- 外部资料型节点：把拆书、知识库、参考作品、用户资料整理成可执行资料包。
- 中枢动作型模块：把某个导演能力包装成创作中枢可调用动作。

每个模块都应通过统一契约说明自己需要读取什么、会产出什么、会影响哪些下游产物、是否可能覆盖用户内容、是否需要审批、失败后如何恢复、是否允许自动重试。这里不需要在方案阶段定义具体字段，但必须先明确契约方向。

新增模块的理想接入方式应是：

1. 在能力注册表中登记模块能力。
2. 声明它属于哪个阶段、读取哪些产物、写入哪些产物。
3. 由 Node Runner 统一执行，而不是让主 service 直接调用内部实现。
4. 由 Policy Engine 决定是否自动执行、暂停确认、允许修复或带风险继续。
5. 由 Artifact Ledger 保存产物版本和依赖。
6. 由事件流向前端、任务中心和创作中枢同步进度。
7. 如果涉及产品级 prompt，按 prompt governance 进入 `server/src/prompting/` 注册。

这样新增功能时，主要工作应是增加一个清晰模块，而不是重新改造整条自动导演链。创作中枢也可以通过能力目录发现可调用动作，避免每新增一个导演能力就额外写一套对话工具分支。

## 7. LangGraph 的定位

LangGraph 适合承担：

- 阶段图编排。
- 条件边。
- interrupt 和人工审批。
- resume。
- 执行追踪。
- 节点级错误处理。

但 LangGraph 不应该承载全部业务复杂度。推荐定位是：

```text
LangGraph
负责“下一步去哪”和“何时暂停/恢复”

Director Runtime
负责“当前导演状态是什么”

Director Node Runner
负责“这个阶段如何执行”

Artifact Ledger
负责“产物如何保存、版本化、比较和恢复”

Policy Engine
负责“是否自动继续、是否需要确认、是否保护用户内容”
```

如果在状态和产物没有统一之前直接 LangGraph 化，风险是把当前两条链路的复杂度搬进图节点，表面上有了图，实际上仍然难以恢复、难以切换、难以解释。

## 8. 模块边界建议

### 8.1 Director Runtime

负责导演运行的生命周期：

- 创建运行。
- 附加已有小说。
- 恢复运行。
- 暂停运行。
- 继续运行。
- 切换控制策略。
- 处理手动编辑后的继续。
- 处理失败恢复。

### 8.2 Workspace Analyzer

负责把当前小说工作区变成导演可理解的生产状态：

- 读取已有小说资产。
- 判断生产阶段。
- 识别缺失与风险。
- 给出推荐下一步。
- 为接管和手动继续提供统一入口。

### 8.3 Artifact Ledger

负责所有导演产物的统一管理：

- 保存产物。
- 标记版本。
- 建立依赖。
- 记录来源。
- 识别过期。
- 支持回滚。
- 支持局部重算。

### 8.4 Policy Engine

负责自动化控制：

- 自动继续。
- 人工确认。
- 自动审批。
- 自动修复。
- 覆盖保护。
- 执行范围。
- 失败处理。
- 质量门分级。
- 单节点重试预算。
- 人工修复和带风险继续。

### 8.5 Capability Registry

负责记录自动导演可用的处理能力，让新增模块能被运行时、阶段图、策略层和创作中枢共同发现：

- 记录模块属于生产、分析、审核、修复、同步、资料接入还是中枢动作。
- 记录模块适用的小说阶段和运行模式。
- 记录模块读取和写入的产物类型。
- 记录模块是否需要审批、是否可能覆盖用户内容、是否允许自动执行。
- 记录模块对应的 PromptAsset、领域服务或外部资料适配器。
- 为创作中枢提供可解释的动作目录，而不是让中枢硬编码每个新能力。

Capability Registry 不应替代 AI 判断。它只是告诉系统“有哪些能力可用、能力边界是什么”；具体何时使用、如何组合、是否推荐给用户，仍应由 AI-first 的工作区分析和下一步决策来完成。

### 8.6 Node Runner

负责执行单个导演阶段：

- 读取所需上下文。
- 调用 PromptAsset 或领域服务。
- 产出结构化产物。
- 写入产物账本。
- 抛出标准事件。
- 返回下一步建议。

Node Runner 应尽量保持通用。新增模块不应要求改 Node Runner 主体逻辑，而应通过能力注册、节点契约、PromptAsset 或领域服务适配接入。

### 8.7 Context Compiler

负责统一上下文组装：

- 根据阶段声明选择上下文。
- 控制 token 预算。
- 压缩长篇连续性信息。
- 注入风格、世界观、角色状态、近期正文、未兑现承诺。
- 保证章节执行不会脱离书级承诺和长期目标。

### 8.8 Source and Knowledge Orchestrator

负责拆书、知识库、续写分析、参考作品分析等资料的导演级接入：

- 判断当前阶段需要哪些资料。
- 提取资料中的结构化决策信息。
- 区分硬约束、软参考、风格参考和禁区。
- 为书级方向、卷规划、章纲和正文生成提供可执行资料包。
- 避免把长资料全文粗暴塞进 prompt，优先提供阶段化、任务化、预算化的知识摘要。

### 8.9 World Architect

负责世界观在自动导演中的生成、绑定、提取和约束：

- 分析已有世界观是否足以支撑当前小说生产。
- 在没有绑定世界观时，按题材需要生成项目级世界观骨架。
- 把世界观转化为冲突规则、行动代价、资源限制、空间结构和信息揭示计划。
- 在卷规划和章节任务单中注入世界规则。
- 检查正文执行是否违反关键设定或削弱世界规则带来的戏剧张力。

### 8.10 Character Governance

负责角色从设定到长篇执行的持续控制：

- 管理主角驱动力、阶段目标、情绪状态和成长方向。
- 管理配角功能、关系推进、出场责任和变化弧。
- 识别角色重复、动机断裂、关系停滞、工具人化等问题。
- 把角色状态注入卷规划、章节任务单和正文执行。
- 在审核与修复阶段输出角色层面的具体修复方向。

## 9. 面向创作中枢的接入方式

后续自动导演应能顺滑接入创作中枢，但接入方式不应是让创作中枢直接调用自动导演内部阶段函数。创作中枢已经有自己的 LangGraph 编排、资源绑定、工具执行、审批 gate 和任务同步节点。自动导演重构后，应作为一个可被创作中枢调用的长运行领域能力，而不是把导演内部图直接揉进创作中枢图里。

### 9.1 当前创作中枢形态

当前创作中枢服务端已经是 LangGraph 风格：

```text
bind_context
  ↓
coordinator_plan
  ↓
tool_execute
  ↓
approval_gate
  ↓
answer_finalize
  ↓
task_sync
```

它适合做会话型协调器：理解用户意图、绑定小说资源、选择工具、处理审批、把执行结果返回消息流。

自动导演则适合做长运行小说生产系统：维护导演状态、阶段产物、恢复游标、自动化策略、章节执行和质量修复。两者应该通过稳定工具和事件协议交互。

### 9.2 接入原则

创作中枢接入自动导演时应遵守以下原则：

- 创作中枢只调用 Director Runtime 的公开动作，不调用内部 phase service。
- 自动导演负责长运行状态、产物账本、恢复、控制策略和执行范围。
- 创作中枢负责会话理解、资源绑定、用户解释、审批呈现和工具结果展示。
- 高风险动作统一走 approval gate，例如覆盖用户内容、重算下游、自动执行大范围章节。
- 自动导演事件应能投影到创作中枢消息流和任务中心，而不是由两边各写一套进度解释。

### 9.3 建议暴露给创作中枢的导演动作

未来可以将自动导演能力以工具形式暴露给创作中枢，建议动作包括：

- `analyze_director_workspace`：分析当前小说状态，返回已有资产、缺失资产、风险、推荐下一步。
- `start_director_run`：从灵感或当前小说状态启动统一导演运行。
- `attach_existing_novel_to_director`：将已有小说接入导演运行时，本质是工作区分析加运行创建。
- `get_director_run_status`：查询当前导演运行、阶段、检查点、等待动作和最近事件。
- `run_director_next_step`：只执行下一步，适合用户想保持手动控制时使用。
- `run_director_until_gate`：自动执行到下一个检查点或风险 gate。
- `switch_director_policy`：切换手动、半自动、自动到检查点、范围自动执行等控制策略。
- `pause_director_run` / `resume_director_run`：暂停或继续导演运行。
- `evaluate_manual_edit_impact`：用户手动修改后，判断哪些产物需要复核、修复或局部重算。
- `explain_director_next_action`：用面向用户的语言解释下一步为什么这样做。

这些动作的返回结果应优先是结构化摘要，而不是大段日志。创作中枢可以基于结构化结果生成消息卡片、审批卡片和后续建议。

后续新增导演模块时，不建议为创作中枢单独写一套分支。更好的方式是让 Capability Registry 产出面向中枢的动作目录：中枢负责理解用户意图、解释动作和承接审批，导演运行时负责实际执行、状态保存、质量门和恢复。这样新增“开篇留存检查”“世界观冲突强化”“角色关系推进诊断”等能力时，只要模块契约稳定，中枢就能以统一方式调用。

### 9.4 事件与审批的衔接

自动导演内部应产出标准 Director Event。创作中枢只订阅或拉取这些事件，并把它们映射为消息流展示和 task sync：

- 阶段开始、阶段完成。
- 产物生成、产物等待确认。
- 工作区分析完成。
- 手动修改影响分析完成。
- 需要审批。
- 自动审批通过。
- 章节执行进度。
- 审核修复进度。
- 失败、暂停、恢复、完成。

审批语义应统一：如果自动导演判断某动作需要人工确认，创作中枢的 `approval_gate` 负责把确认卡片展示给用户；用户确认后，创作中枢再调用导演运行时继续，而不是自己绕过导演策略直接执行底层服务。

### 9.5 接入迁移顺序

推荐迁移顺序：

1. 先开放只读工具：工作区分析、导演状态、下一步解释。
2. 再开放低风险动作：只运行下一步、暂停、继续。
3. 再开放策略切换：手动、半自动、自动到检查点。
4. 最后开放高风险动作：范围自动执行、重算下游、覆盖或修复已有产物。

这样创作中枢可以先成为自动导演的“解释与控制入口”，再逐步成为完整执行入口。用户体验上会更自然：用户可以在创作中枢里问“现在该怎么办”“继续自动导演到下一步”“我改了主角动机，帮我看看后续要不要重算”，系统直接调用统一导演运行时处理。

## 10. 用户手动编辑后的继续机制

这是本次重构最重要的体验改进点之一。

当用户手动编辑小说资产后，系统不应该简单把当前自动任务标记为中断，也不应该要求用户手动选择复杂恢复策略。系统应该执行以下流程：

1. 记录用户修改对应的产物版本。
2. 分析该修改可能影响哪些下游产物。
3. 判断是否可以直接继续。
4. 如果不能直接继续，给出最小修复路径。
5. 在自动覆盖或重算前请求确认。

例如：

- 用户修改主角动机后，角色关系、卷目标和后续章节任务单可能需要复核。
- 用户只润色某一章正文时，通常不需要重做宏观规划，但可能需要更新连续性记忆。
- 用户删除重要伏笔时，系统应提示后续 payoff 需要调整。
- 用户改动卷结尾目标时，后续章节任务单需要局部重算。

这些判断应尽量由 AI 结构化输出，并由确定性代码执行保护。

质量审核或承诺校验失败时，也应使用同一套继续机制。系统应说明失败影响的是哪个产物、哪个章节窗口或哪个后续承诺，而不是只给出“流程失败”。用户可以选择让系统执行一次自动修复、自己手动修复后继续，或在明确风险后先推进后续步骤。

## 11. 面向新手完成整本书的设计原则

自动导演重构不只是工程清理，也应该服务产品目标：帮助完全写作新手完成一部长篇小说。

因此，新架构应坚持：

- 默认给推荐动作，而不是让用户选择复杂流程参数。
- 每个阶段只让用户做少量高价值判断。
- 所有专业判断都尽量由 AI 给出解释和建议。
- 章节执行必须始终回到书级承诺、卷目标、角色动机和章节钩子。
- 审核不只指出问题，还要给出可执行修复方案。
- 自动化范围越大，越要清楚展示影响范围和退出点。
- 用户手写内容必须被尊重和保护。
- 失败恢复要像正常流程的一部分，而不是异常补丁。

## 12. 渐进迁移路线

### Phase 1：方案冻结与现状对账

目标：

- 梳理当前自动导演创建、接管、继续、恢复、自动执行的真实链路。
- 标出状态来源、产物来源、检查点来源和重复判断。
- 确认哪些能力必须保留，哪些只是历史兼容。

交付：

- 当前链路对账表。
- 统一运行时边界文档。
- 需要迁移的旧入口清单。
- 拆书、知识库、世界观、角色资产在当前自动导演链路中的参与度对账。

### Phase 2：引入统一运行时外壳

目标：

- 新建统一 Director Runtime 门面。
- 保留现有外部 API。
- 让新建、接管、继续都先进入统一运行时，再转调旧实现。

交付：

- 统一入口。
- 统一运行状态快照。
- 统一事件模型草案。
- 初版能力注册表和节点契约草案。

### Phase 3：接管链路并入工作区分析器

目标：

- 把接管逻辑从独立恢复链路改成工作区分析。
- 让已有小说、新建小说中断恢复、用户手动修改后继续，都通过同一套分析结果进入导演图。

交付：

- 工作区分析器。
- 推荐下一步机制。
- 手动编辑影响分析的第一版。
- 是否需要生成或补齐项目级世界观的判断机制。

### Phase 4：产物账本落地

目标：

- 把关键导演产物统一纳入版本化账本。
- 逐步减少 seed payload 和 checkpoint 对业务真实状态的承载。

交付：

- 核心产物的统一读写入口。
- 产物版本和依赖关系。
- 下游过期标记机制。
- 拆书摘要、知识资料包、世界观骨架、角色治理状态进入统一产物体系。

### Phase 5：控制策略层落地

目标：

- 把自动、手动、半自动、自动审批、自动修复从各阶段分支中抽出来。
- 让用户可以在同一运行中切换控制权。

交付：

- 统一控制策略。
- 暂停、继续、只运行下一步、自动运行到检查点等动作。
- 覆盖保护和确认机制。
- 质量门分级、一次自动重试、人工修复和带风险继续动作。

### Phase 6：阶段图化

目标：

- 在状态、产物、策略边界稳定后，将阶段推进改为显式图。
- 可选择使用 LangGraph 承载 interrupt、resume、trace。

交付：

- 统一阶段图。
- 节点契约。
- 可插拔节点发现和装配机制。
- 图运行追踪。
- 可恢复节点。
- 至少选择一个低风险新模块作为扩展样例，验证新增模块不需要改主链。

### Phase 7：旧链路收口

目标：

- 移除或下沉旧的接管专用分支。
- 简化大型 service。
- 保证所有入口都走统一运行时。

交付：

- 自动导演创建和接管共用一条主链。
- 手动/自动切换稳定。
- 任务中心、前端进度和后端事件一致。

## 13. 风险与约束

### 13.1 数据安全

重构过程中不得执行破坏性数据操作。涉及迁移、重算、删除旧产物或覆盖用户内容时，必须遵守项目数据保护规则：

- 明确用户批准。
- 有可验证备份。
- 有恢复验证或至少备份存在性与大小检查。

### 13.2 AI-first 约束

不得用关键词、硬编码规则、正则分支来替代核心意图识别、接管判断、改动影响判断和下一步推荐。

允许确定性逻辑处理：

- 输入校验。
- 安全保护。
- 幂等控制。
- 数据范围检查。
- 对 AI 结构化输出的后处理。

### 13.3 兼容性

迁移期间需要保证：

- 现有自动导演入口可继续使用。
- 现有任务中心状态不突然失效。
- 历史任务尽量能读取和解释。
- 失败恢复路径不能被一次性切断。

### 13.4 文件复杂度

自动导演核心文件已经超过适宜维护规模。后续实现时应强制模块化，避免继续把编排、状态、恢复、接管、策略和 UI 解释堆在同一个 service 内。

## 14. 验收标准

本次重构最终应达到以下效果：

- 新建小说和接管已有小说使用同一个导演运行时。
- 用户可以在同一条运行链中从自动切到手动，再从手动切回自动。
- 接管已有小说时，系统能先给出可理解的下一步推荐，而不是暴露工程化阶段参数。
- 手动编辑后，系统能判断是否需要补齐、复核、局部重算或直接继续。
- 拆书和知识库能深入参与书级方向、卷规划、章节任务单和正文执行，而不是只作为外围参考。
- 未绑定已有世界观时，系统能按题材判断是否需要生成项目级世界观骨架，并在后续规划和正文中持续使用。
- 角色控制能贯穿宏观规划、卷规划、章节任务和审核修复，而不是停留在一次性角色卡生成。
- 自动执行范围、审批点、修复策略和覆盖保护由统一控制策略管理。
- 关键产物有统一账本，不再主要依赖任务 payload 拼接真实业务状态。
- 前端进度、任务中心、后端检查点和导演事件表达一致。
- 失败恢复、服务重启后的待恢复提示和用户主动继续走同一套恢复语义。
- 单个审核失败、承诺校验失败或章节修复失败不会冻结整条导演链。
- 自动修复默认最多重试一次，失败后允许人工修复、接受风险继续或后续局部重算。
- 质量问题会记录受影响范围、修复建议和继续策略，而不是只返回全局失败。
- 新增一个生产、分析、审核、修复或同步模块时，可以通过能力注册、节点契约、产物账本和策略层接入，不需要新增一条独立链路。
- 新增导演能力后，创作中枢可以通过统一动作目录调用，而不是为每个能力写独立工具分支。
- LangGraph 如被引入，只负责编排、暂停、恢复和追踪，不吞掉领域边界。

## 15. 当前阶段不做的事

为避免方案阶段过早固化实现，本阶段不定义：

- 具体数据库字段。
- 具体 API 请求/响应字段。
- 具体表结构。
- 具体前端组件拆分。
- 具体 PromptAsset 内容。
- 具体 LangGraph 节点代码。

这些内容应在方案边界确认后，结合现有代码分阶段设计和迁移。

## 16. 一句话方向

自动导演不应再分为“自动创建链”和“AI 接管链”。目标是建立一个统一的小说生产导演运行时：它能从空白灵感开始，也能接管已有小说；能自动推进，也能尊重用户手动创作；能暂停、恢复、局部修复，并始终围绕帮助新手完成整本小说来组织下一步。

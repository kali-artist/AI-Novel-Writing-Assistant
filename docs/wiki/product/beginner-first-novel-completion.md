# 新手优先与整本小说完成原则

## 背景

项目的主要用户是完全写作新手。他们通常不知道如何设计结构、节奏、角色弧、伏笔、卷纲或章节任务单。产品不能假设用户能手动修复长篇结构问题，也不能把复杂专业判断抛给用户。

因此，产品目标不是提供零散写作工具，而是帮助用户从模糊灵感持续推进到完整小说。

## 决策

所有产品、UX、Prompt、Agent 和 runtime 决策优先服务“新手完成整本小说”。当专家灵活性和新手完成率冲突时，默认选择降低认知负担、给出强默认值和明确下一步的方案。

AI 应承担规划、判断、调度、执行、追踪和修复建议；确定性代码负责安全、幂等、输入校验、权限、状态落库和结构化输出后的处理。

## 当前规则

- 关键页面应告诉用户当前做到哪里、下一步是什么、为什么推荐这一步、风险影响哪些范围。
- 自动导演、章节生产和 Creative Hub 都应围绕整本完成主链，不新增无助于主链的复杂分支。
- 产品默认值应能让新手继续前进；高级选项不能阻塞基础流程。
- 失败后应提供可恢复路径、局部影响范围和下一步建议，不只显示错误。
- UI 文案从用户视角写功能和下一步，不写实现迁移、架构变化或“我们改了什么”。
- 质量判断、下一步推荐和修复路径应通过 AI-first 结构化理解完成。

## 示例

推荐做法：

- 自动导演面板显示当前阶段、等待确认点、可自动审批项和写入范围。
- 章节执行显示“正文可读”“资产回灌中”“账本校准中”等不同状态，让用户先看正文，再观察后台同步。
- Creative Hub 回答小说进展时，先说明真实产物进度，再补充后台任务情况和推荐下一步。

禁止做法：

- 要求新手自己判断是否应该重规划、修章节、重跑角色或修改世界观。
- 给用户展示内部重构、迁移或历史实现叙述作为产品说明。
- 把 Creative Hub 扩展成泛聊天工具，而不推进小说完成。

## 失败模式

- 用户看到很多按钮但不知道下一步：检查是否缺少单一推荐动作和原因。
- 用户以为任务卡住：检查前端是否展示真实产物进度、阻塞原因和恢复动作。
- AI 判断失败后加了关键词兜底：应修 AI schema、prompt、上下文和评估，而不是让产品逻辑退回硬编码。
